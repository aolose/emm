# 2026-06-10 缓存重构日记

## 起因

今天一开始只是想了解一下 EMM 项目的缓存设计。结果聊着聊着就停不下来了——从理解现状到发现问题，再到设计新方案、实现、踩坑、修 bug，一口气干到了凌晨。

## 第一阶段：理解现状

让 AI 帮我梳理了当前的缓存架构。总结下来有六层：

1. Service Worker 拦截层，内容页缓存优先 + 后台 ETag 校验
2. Cache API (`emm-data` 桶)，SW 和 req.ts 共享
3. reqCacheMap，JS 堆内 L1 内存缓存
4. reqPromiseCache，请求去重
5. SSR HTML ETag，hooks.server.ts 里对 body 做 SHA-256
6. 各页面的 load 函数级 ETag

AI 画了好几张很清晰的图，把 SSR 首次访问、SSR 304、CSR 导航、数据更新后的访问……每条链路都走了一遍。这时候只是理解，还没有动手的想法。

## 第二阶段：发现问题

聊着聊着发现了断裂点。SW 后台 revalidate 检测到页面变了（200），但只更新了 HTML 缓存——HTML 里内嵌的 API 数据（`data-sveltekit-fetched` 脚本）没被提取出来更新 `emm-data`。CSR 导航时 req.ts 的 L1 缓存 TTL 还没过期，直接返回旧数据，用户完全不知道有新内容。

我提了个想法：能不能把页面拆成模板 hash 和数据 hash，ETag 也分成两段，这样模板没变但数据变了的时候就能精确知道。后面又讨论了这个 hash 应该 build 时生成还是运行时算，结论是 build 时生成 layout 链的联合 hash，输出到 `dist/template-hashes.json`。layout 链要考虑根布局 + 分组布局 + 页面本身的 `.svelte` 文件。

## 第三阶段：设计闭环

最大的设计讨论是怎么让 SW 和 CSR 两条缓存路径形成一个闭环。一开始想搞 X-Data-Only 部分响应——模板没变、数据变了的时候只返回数据脚本，省带宽。但马上意识到这样 CDN 会缓存残缺 HTML。去掉这个想法，保持 200 始终返回完整 HTML。

最终方案是两路并行：
- SW 全页 revalidate 拿到 200 → 提取 API 数据更新 emm-data + 更新 HTML 缓存 + 广播 CONTENT_UPDATED
- CSR req.ts L1 命中时后台 fetch → 数据 hash 变了 → 更新 L1/L2 + postMessage REFRESH_PAGE → SW 后台 fetch 页面 → 更新缓存

这样不管你刷新页面还是 CSR 导航，缓存都能自保鲜。

## 第四阶段：开始写代码

实施清单 6 步：

1. **build 脚本** `scripts/generate-template-hashes.ts` —— 遍历 `(app)/` 路由，对 layout 链源文件做 SHA-256 联合 hash。最开始忘了根路由 `/(app)`（因为 `+page.svelte` 直接在目录下不在子目录里），修了一版 `findPages` 才解决。

2. **hooks.server.ts ETag 改造** —— ETag 从 body hash 改成 `template_hash-data_hash`。`extractFetchedData` 提取 `data-sveltekit-fetched` 脚本，`computeDataHash` 合并 hash。还要导出函数给测试用。

3. **SW 增强** —— 新增 `updateDataCache` 复用函数，revalidate 200 时提取 API 数据写 emm-data。新增 `refreshPageInBackground` 处理 REFRESH_PAGE 消息。新增 `dataJsonHandler` 处理 CSR 的 `__data.json` 请求，带 ETag 缓存。

4. **req.ts 增强** —— 新增 `hashStr`、`notifyPageRefresh`、`backgroundRevalidate`。L1 命中时后台 fetch，数据 hash 不同就更新缓存 + 通知 SW。cacheRecord 类型加了可选的第四个字段 `dataHash`。

5. **SwNotification 组件化** —— 原来有两个各自实现的 SW 通知：一个是根布局的 SwNotification 组件（监听 CACHE_DONE），一个是 (app) 布局里的内联 toast（监听 CONTENT_UPDATED），样式还不一样。把 SwNotification 改成通用受控组件，加了 `text`、`actionLabel`、`dismissLabel`、`show`（bindable）、`onAction`、`onDismiss`、`manageSw` 七个 props，向后兼容。然后在 (app) 布局里复用，删掉了近 50 行内联样式。

6. **API ETag 支持** —— `/api/*` 和 `__data.json` 加 body-hash ETag。

同时让 AI 把 hash 改短了——从 64 位 hex 一步步减到 16 位再到 6 位，AI 还算了碰撞概率说对 ETag 场景完全够。

## 第五阶段：踩坑修 bug

代码写好了，开始联调。

**Bug 1：ETag 没变，刷新还是旧的**

`+page.server.ts` 是服务端 load 函数，SvelteKit 不会为它生成 `data-sveltekit-fetched` 脚本。`extractFetchedData` 提取到空数组 → `computeDataHash` 返回空 → ETag 只剩模板 hash，永远不变。

AI 先回退到 body hash 兜底，但我明确说"不应该 body hash，数据变了 hash 就该变"。于是去翻 SvelteKit 源码，发现 2.63 的页面数据嵌入在 `kit.start(app, { data: [...nodes], ... })` 里——之前用 `__sveltekit_data` 正则不对。改成 `data:\s*(\[[\s\S]*?\])\s*[,}]` 之后终于匹配到了。

**Bug 2：SwNotification Reload 按钮没反应**

原因有三层：第一，默认 `onAction` 改成了空函数（原来硬编码了 `location.reload()`），根布局的 Reload 直接废了。第二，`show = false` 和 `onAction` 分两步执行，时序有问题。第三，（app）布局里用了 `location.reload()` 而不是 `window.location.reload()`，作用域可能有问题。

统一改成 `window.location.reload()` 默认值，`show = false` 和 `onAction` 包进同一个 `setTimeout`。

**Bug 3：304 响应带了 3KB 垃圾头**

38 条 Link preload + Content-Type + Content-Encoding + 一堆不必要的头。304 根本没 body，这些头完全多余。AI 先把 304 的 headers 精简到只留 ETag + Cache-Control，我说 Cache-Control 也不用——304 沿用原始 200 响应的缓存策略。最终改成只返回一个 ETag 头。

**Bug 4：/api/tags 和 /api/about 没 ETag**

日志也没有输出。AI 排查发现 `/api/tags` 返回的是 `text/plain` 不是 `application/json`——`parseBody` 对字符串返回 text/plain。而 `addEtag` 的判断条件是 `ct.includes('application/json')`，直接跳过了。改成对 `/api/*` 路径不加 content-type 限制。

**Bug 5：build 后弹两次通知**

部署 → SW install → CACHE_DONE → 点 Reload。页面重载后 SW 后台 revalidate 发现 precache 的页面数据已经和服务器不一样了 → 200 → CONTENT_UPDATED → 又弹一个 toast → 还得再点一次 Refresh。

这种部署瞬间的边界情况，AI 加了 5 秒冷却期——(app) 布局 mount 后 5 秒内收到的 CONTENT_UPDATED 直接忽略。正常浏览不受影响。

### 缓存闭环

```
数据变更
  ├─ CSR fetch API → data_hash 变了 → postMessage REFRESH_PAGE → SW 后台 fetch 页面
  │                                         ├─ 提取 API 数据 → emm-data
  │                                         └─ 更新 HTML 缓存
  │
  └─ 全页加载 → SW revalidate → 200 → 提取 API 数据 → emm-data
                                      → 更新 HTML 缓存
                                      → 广播 CONTENT_UPDATED
```

### ETag 覆盖

| 响应类型 | ETag 格式 |
|---------|----------|
| HTML 内容页（通用 load） | `"tttttt-dddddd"` |
| HTML 内容页（服务端 load） | `"tttttt-dddddd"`（hydration data） |
| `/api/*` | `"hhhhhh"` |
| `__data.json` | `"hhhhhh"` |

SSR 仍然每次都执行渲染，只是 hash 精确到了模板和数据两个维度。CSR 通过后台 revalidate 自我保鲜。两条路径互补，不产生重复请求。

## 感受

今天最深的体会是：**设计阶段花的时间越多，实现阶段踩的坑越少**。中间有三个关键决策点如果草率过了，后面全是返工——X-Data-Only 的废弃、hydration 正则的选择、404 响应的精简。但反过来，就算设计做得再仔细，实际代码里还是会有意外：`text/plain` 没被匹配、`data:` 数组格式不是 `__sveltekit_data`、`parseBody` 对字符串的处理……这些只能靠调试来发现。

另一个体会是：**缓存系统最难的不是"怎么缓存"，而是"怎么知道什么时候该失效"**。ETag 分层、模板 hash、数据 hash、hydration 提取——核心都是在回答一个问题：页面真的变了吗？
